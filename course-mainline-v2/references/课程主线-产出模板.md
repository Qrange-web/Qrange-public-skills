# 课程主线产出模板

> [!info] 用途
> 整理课程主线：把课程讲稿、逐字稿、书籍原文等任何结构化材料加工成一份标准化课程主文档。
>
> - **输入**：用户提供的任意材料（逐字稿、书籍原文、PPT文本、论文等）
> - **输出**：9 模块结构化 Markdown 文档

---

## 硬约束（必须遵守）

### Frontmatter

```yaml
---
type: 课程主线
skill_name: <技能名称>
date: YYYY-MM-DD
source:
  - "<源材料1>"
  - "<源材料2>"
status: 已完成
---
```

### 九模块骨架（一个不能少）

```
# <技能名称> — 课程主线

## 1. 课程一句话定位
## 2. 课程要解决的核心问题
## 3. 核心方法
## 4. 操作流程
## 5. 判断标准
## 6. 适用场景与边界
## 7. 常见误区
## 8. 需要谨慎处理的讲师观点
## 9. 资料来源说明
```

### 五个格式规则

1. 模块编号用阿拉伯数字 `1. 2. 3. ...`
2. `**` 不包标点——正确：`**核心概念**：说明`，错误：`**核心概念：**说明`
3. 第 5 节判断标准统一用三列表格：`检验项 | 通过标准 | 不通过时`
4. 第 7 节每条误区必须配课程原话纠正，不自己发明
5. 第 8 节讲师观点必须标注类型（核心主张/金句/经验主张/趋势判断）

---

## 软约束（建议遵守，但不强制）

### 每个模块要回答的问题

| 模块 | 回答这个问题 | 易犯错误 |
|------|------------|---------|
| 1. 定位 | 这门课到底在教什么？ | 写成课程简介而非方法论定位 |
| 2. 核心问题 | 不学这门课会犯什么错？ | 痛点列了一堆但没有收敛到一句话 |
| 3. 核心方法 | 这门课的方法论骨架长什么样？ | 面面俱到堆砌知识点，没有主次结构 |
| 4. 操作流程 | 学完之后第一步做什么、第二步做什么？ | 把第 3 节换个说法再说一遍 |
| 5. 判断标准 | 怎么判断自己做对了还是做错了？ | 只写"好的标准"没写"差的信号" |
| 6. 适用边界 | 什么场景用它，什么场景别用？ | 适用写太多，不适用写太少 |
| 7. 常见误区 | 新人最容易踩的坑是什么？ | 自己编纠正，不用课程原话 |
| 8. 讲师观点 | 哪些是讲师的主张 vs 可验证的事实？ | 把讲师观点当客观规律写进方法论 |
| 9. 资料来源 | 原材料是什么？各自贡献了什么？ | 只列文件名不写角色 |

### 第 3 节（核心方法）的展开方式

先用一个概述表给出全貌，然后按课程的知识模块逐个展开为 3.1、3.2、3.3... 子节。

> [!example] 子节怎么写、写几个、用什么节奏——取决于课程内容本身，不强求统一格式

以下是常见的展开方式，按需选用：

| 内容复杂度 | 展开方式 |
|-----------|---------|
| **复杂子方法论** | 引用块定性 → 核心原则 → 工具表 → 实战心法（最完整） |
| **中等复杂度** | 一句话概括 + 表格（最常见） |
| **轻量补充信息** | 一个表格足够（如段位表、公式表） |
| **纯流程** | ASCII 流程图 |

### 第 4 节（操作流程）的建议

优先用 ASCII 流程图，比纯文字清晰。有分支路径时先做入口判断再分流。

### 引用块和粗体的惯例

- 课程核心主张、金句、一段概括用 `>` 引用
- 核心概念首次出现加粗
- 关键结论句加粗

### 文档长度

不设限制——课程方法简单就短，复杂就长，以方法论提取完整为准。
